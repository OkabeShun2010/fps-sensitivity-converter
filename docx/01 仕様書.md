# FPS感度変換ツール 仕様書

## 1. プロジェクト概要

### 1.1 プロジェクト名

FPS感度変換ツール

### 1.2 目的

VALORANT、Apex Legends、Overwatch 2 の通常感度を、同じ `cm/180` になるように相互変換できるWebアプリを作成する。

`cm/180` は、ゲーム内で視点を180度振り向かせるために必要なマウス移動距離を指す。

### 1.3 想定ユーザー

- 複数のFPSタイトルをプレイしており、ゲーム間で近い操作感を再現したいユーザー
- DPI、ゲーム内感度、eDPI、`cm/180` の関係を確認したいユーザー
- ブラウザ上で素早く感度を計算したいユーザー

## 2. MVPスコープ

### 2.1 対応ゲーム

初期版では以下の3タイトルに対応する。

- VALORANT
- Apex Legends
- Overwatch 2

### 2.2 対応する感度

初期版では通常感度のみを対象とする。

以下は初期版では対応しない。

- ADS感度
- スコープ感度
- スコープ倍率ごとの感度
- 視野角による体感差の補正
- ヒーロー、キャラクター、武器ごとの個別設定

### 2.3 アプリ形態

- ブラウザで動作するWebアプリ
- バックエンドなし
- ログインなし
- データベースなし
- 外部APIなし

## 3. 機能要件

### 3.1 入力項目

| 項目 | 内容 | 必須 |
| --- | --- | --- |
| 変換元ゲーム | 現在使っている感度のゲーム | 必須 |
| 変換先ゲーム | 感度を変換したいゲーム | 必須 |
| ゲーム内感度 | 変換元ゲームで設定している通常感度 | 必須 |
| DPI | マウスDPI | 必須 |

### 3.2 出力項目

| 項目 | 内容 |
| --- | --- |
| 変換後感度 | 変換先ゲームで設定すべき通常感度 |
| 変換元eDPI | `DPI * 変換元感度` |
| 変換先eDPI | `DPI * 変換後感度` |
| cm/180 | 180度振り向くために必要なマウス移動距離 |
| 計算説明 | 変換元と変換先のゲーム名、感度、`cm/180` を使った簡単な説明 |

### 3.3 基本操作

1. ユーザーが変換元ゲームを選択する。
2. ユーザーが変換先ゲームを選択する。
3. ユーザーがゲーム内感度を入力する。
4. ユーザーがDPIを入力する。
5. アプリが変換後感度、eDPI、`cm/180` を表示する。

### 3.4 同一ゲーム選択時の挙動

変換元ゲームと変換先ゲームが同じ場合、変換後感度は入力したゲーム内感度と同じ値を表示する。

## 4. 感度変換ロジック

### 4.1 基本方針

ゲームごとの `yaw` 値を使い、同じ `cm/180` になるように感度を変換する。

`yaw` は、マウス入力1カウントあたりにゲーム内視点が何度回転するかを表す値とする。

### 4.2 使用する係数

初期版では以下の係数を使用する。

| ゲーム | yaw |
| --- | ---: |
| VALORANT | 0.07 |
| Apex Legends | 0.022 |
| Overwatch 2 | 0.0066 |

係数は実装時に定数として管理する。

### 4.3 cm/180 の計算式

```text
cm/180 = 180 / (DPI * ゲーム内感度 * yaw) * 2.54
```

### 4.4 変換後感度の計算式

同じDPIで変換する場合、変換後感度は以下の式で求める。

```text
変換後感度 = 変換元感度 * 変換元ゲームのyaw / 変換先ゲームのyaw
```

### 4.5 eDPI の計算式

```text
eDPI = DPI * ゲーム内感度
```

eDPIはゲーム内感度とDPIを掛けた値であり、ゲームごとの `yaw` が異なるため、異なるゲーム間で単純比較できる値ではない。

### 4.6 小数点の扱い

- 変換後感度は小数第4位まで表示する。
- `cm/180` は小数第2位まで表示する。
- eDPIは小数第2位まで表示する。
- 内部計算では丸めず、表示時のみ丸める。

## 5. バリデーション

### 5.1 入力値の制約

| 項目 | 制約 |
| --- | --- |
| ゲーム内感度 | 0より大きい数値 |
| DPI | 0より大きい数値 |
| 変換元ゲーム | 対応ゲームから選択 |
| 変換先ゲーム | 対応ゲームから選択 |

### 5.2 エラー表示

以下の場合は結果を表示せず、入力欄の近くにエラーメッセージを表示する。

- 感度が未入力
- 感度が0以下
- DPIが未入力
- DPIが0以下
- 数値として解釈できない値が入力された

## 6. UI/UX要件

### 6.1 デザイン方針

- ダークテーマを基本とする。
- シンプルなカード型レイアウトにする。
- 1画面で入力から結果確認まで完結できる構成にする。
- ゲーミング風に寄せすぎず、読みやすさを優先する。
- 公式ロゴや商標素材は初期版では使用しない。

### 6.2 画面構成

初期版は1ページ構成とする。

- ヘッダー
  - アプリ名
  - 簡単な説明文
- 入力カード
  - 変換元ゲーム
  - 変換先ゲーム
  - ゲーム内感度
  - DPI
- 結果カード
  - 変換後感度
  - 変換元eDPI
  - 変換先eDPI
  - `cm/180`
  - 計算説明
- 補足説明
  - `cm/180` の意味
  - eDPIの注意点

### 6.3 レスポンシブ対応

- PCとスマートフォンの両方で操作できるようにする。
- スマートフォンでは入力項目と結果を縦並びにする。
- PCでは入力カードと結果カードを横並びにしてもよい。

## 7. 非機能要件

### 7.1 パフォーマンス

- 入力変更後、即座に結果を再計算する。
- 外部API通信を行わないため、オフラインに近い体験で動作する。

### 7.2 アクセシビリティ

- 入力欄にはラベルを付ける。
- エラー文は入力欄と関連付けて表示する。
- 色だけに依存せず、テキストでも状態を伝える。

### 7.3 保守性

- 計算ロジックはUIから分離する。
- ゲームごとの係数は配列またはオブジェクトの定数として管理する。
- 将来的なゲーム追加時に、UIと計算式の大幅変更が不要な構成にする。

## 9. データ設計

### 9.1 ゲーム定義

ゲームごとの情報は、実装時に以下のようなデータ構造で管理する。

```ts
type Game = {
  id: string;
  name: string;
  yaw: number;
};
```

### 9.2 ゲームデータ例

```ts
const games: Game[] = [
  { id: "valorant", name: "VALORANT", yaw: 0.07 },
  { id: "apex-legends", name: "Apex Legends", yaw: 0.022 },
  { id: "overwatch-2", name: "Overwatch 2", yaw: 0.0066 },
];
```

## 10. テスト観点

### 10.1 計算ロジック

- VALORANTからApex Legendsへ変換できること
- Apex LegendsからVALORANTへ変換できること
- Overwatch 2からVALORANTへ変換できること
- 同一ゲームを選択した場合、入力感度と同じ値になること
- DPIが変わっても、同じDPI内の変換後感度は `yaw` 比率で決まること
- `cm/180` が正しく計算されること

### 10.2 入力バリデーション

- 感度が空の場合にエラーになること
- 感度が0以下の場合にエラーになること
- DPIが空の場合にエラーになること
- DPIが0以下の場合にエラーになること
- 数値以外の入力でエラーになること

### 10.3 UI

- PC幅で入力と結果が見やすく表示されること
- スマートフォン幅で縦並びに表示されること
- エラー時に結果が表示されないこと
- 入力変更時に結果が更新されること
